# 趋势功能与 Resilience Index 决策清单

这份文件用来帮助团队确认两个还没有正式落地的部分：

1. 趋势功能（Historical Trends）
2. Resilience Index

目前项目已经完成真实数据桥接与前端替换 mock data 的主线工作，但这两个部分仍然需要团队先统一范围与做法，才适合继续实现。

---

## 一、趋势功能（Historical Trends）

### 现在的状态

- 当前 dashboard 以 `snapshot` 为主。
- 目前更适合显示“最新一笔指标值”的比较结果。
- 前端已经明确把趋势功能视为未正式启用。
- 如果要做真实趋势，底层数据结构大概率需要升级。

### 需要讨论的问题

1. 这次交付要不要做真实趋势？
2. 如果要做，只做少数关键指标，还是主要指标都做？
3. 目前现有资料是否真的足够支撑多年份趋势？
4. 数据库是否要升级成能保存 `(territory, indicator, year)` 的结构？
5. 如果这次不做，前端是继续标记为 `snapshot only`，还是直接隐藏趋势区域？

### 可选方案

#### 方案 A：本阶段不做真实趋势

适合情况：

- 时间有限
- 团队现在优先目标是完成 handoff 主线
- 目前数据库结构还不是完整 time-series 架构

优点：

- 风险最低
- 不需要大改数据库结构
- 可以避免做一半又推翻

缺点：

- 趋势图暂时不能算正式完成

#### 方案 B：本阶段开始做真实趋势

适合情况：

- 团队确认这次就必须展示年度变化
- 数据层愿意同步改 schema
- 有时间重新验证历史序列逻辑

优点：

- 功能更完整
- 可以支持真正的年度趋势图

缺点：

- 需要改数据库结构
- 风险高
- 容易影响现有 snapshot 稳定性

### 建议结论

建议先采用：

> 本阶段先完成真实数据的 snapshot dashboard，趋势功能列为下一阶段，待 schema 升级后再实现。

---

## 二、Resilience Index

### 现在的状态

- 目前系统已经可以展示真实指标。
- 但 `Resilience Index` 还不是单纯把数据接上就会自动完成的功能。
- 它需要明确的计算方法、缺值处理规则，以及展示方式。

### 需要讨论的问题

1. 这次交付要不要正式做 Resilience Index？
2. 如果要做，是否完全采用现有 methodology 文件？
3. 分数是要做总分，还是总分加各 pillar 分数？
4. 指标缺值时怎么处理？
5. 谁负责实现计算逻辑，谁负责前端展示？

### 可选方案

#### 方案 A：本阶段先不做正式 Resilience Index

适合情况：

- 团队目前先确保真实指标展示完成
- 指数算法还没有完全对齐
- 不想因为公式或缺值处理争议影响交付

优点：

- 范围清楚
- 可以避免算法解释不一致
- 更容易保证当前成果稳定

缺点：

- 没有 headline score

#### 方案 B：本阶段正式实现 Resilience Index

适合情况：

- 团队已经确认 methodology 不再改
- 缺值处理规则也已经拍板
- 有时间做额外验证

优点：

- 页面更完整
- 更接近最终分析型产品

缺点：

- 需要额外实现和验证
- 缺值与权重问题很容易被问

### 建议结论

建议先采用：

> 本阶段先完成真实指标 dashboard；Resilience Index 待 calculation method 与 missing-data rules 完全确认后再实现。

## 五、当前推荐执行方向

如果团队希望先稳住交付，建议顺序如下：

1. 先确认 handoff 主线完成
2. 先以 snapshot dashboard 作为当前成果
3. 趋势功能先不强行上线
4. Resilience Index 先不在本阶段正式落地
5. 下一阶段再单独规划 schema 升级与指数计算
