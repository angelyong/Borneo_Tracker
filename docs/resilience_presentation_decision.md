# Resilience 呈现方式 — 决策简报（待与 Supervisor 讨论）

> **状态：** 待决策 · 不实现任何东西，先定方向
> **日期：** 2026-07-09
> **要问 Supervisor 的一句话：**
> **「Resilience（韧性）这个东西，应该继续用『六角形图 + 0–100 评分』呈现给用户，还是转换成一种以场景/故事为主、把评分藏到后台的方式？」**

---

## 1. 背景：Resilience 现在是什么

Resilience Index 是一个**合成的复合分数**（`compute_resilience.py`）：

1. 每个可打分指标按 documented bounds 线性归一化到 **0–100**；
2. **支柱分 = 该支柱下已打分指标的平均**（True Wealth 六角：Food / Energy / Education / Shelter / Healthcare / Entertainment）；
3. **韧性指数 = 有分数的支柱的平均** → 映射成红/黄/绿。

当前四地区的分数（`public/data/resilience.json`）：

| 地区 | 指数 | 灯 | 已打分支柱 |
|---|---|---|---|
| Brunei | 81.2 | 🟢 | Education, Energy, Healthcare |
| Kalimantan | 69.2 | 🟢 | Education, Energy, Healthcare |
| Sarawak | 58.6 | 🟡 | Education, Healthcare |
| Sabah | 56.9 | 🟡 | Education, Healthcare |

---

## 2. 发现的问题（为什么要开这个决策）

**从用户角度看，一个裸露的评分几乎没有意义。**

- 用户看到「韧性 56.9 / 黄灯」，会想「56.9 是高是低？黄灯又怎样？关我什么事？」然后划走。
- 要看懂这个分数，用户得先理解归一化方法、六角支柱、bounds 怎么定——**没人会为了看个仪表盘去读方法论**。
- 评分回答的是「专家怎么打分」，没回答用户真正在问的：**「我的地方还好吗？」「在变好还是变坏？」「这跟我有什么关系、要我做什么？」**

**同时，评分本身有几个已知弱点，让它更不适合直接当门面：**

- 现在每个地区只算得出 **2–3 根支柱**（六角里其余 3–4 根没有分数）——是个覆盖不全的半成品。
- 归一化的 best/worst bounds 是原方法论文档被删后**重建**的，尚未校准，绝对分值偏主观。
- 单一评分**会误导**：例如 Brunei 总分 81.2 绿灯，但它本地稻米年产仅 **3,700 吨**（vs Sabah 107,565 吨），一旦食物断供其实很脆弱——绿灯把这个风险藏起来了。

---

## 3. 三条可选路径

### 路径 A — 维持现状：六角形图 + 评分
- **做法：** 前台直接展示 True Wealth 六角雷达图 + 各支柱分 + 总指数 + 红黄绿。
- **优点：** 已经做好、信息密度高、对专业/学术受众（评审、政策）一目了然；契合方法论。
- **缺点：** 对普通公众认知门槛高、意义不明；半成品支柱和重建 bounds 的弱点会直接暴露在门面上。
- **适合：** 如果这个产品的主要受众是**评审 / 政策 / 研究者**，而非大众。

### 路径 B — 转换呈现：把评分变成「场景 / 影响」，评分退到后台
- **做法：** 前台不显示 0–100 分，而是用**真实数据讲影响**。已做了一个可点样板「**Resilience Stress Test（压力测试）**」：
  用户选地区 + 选冲击（食物断供 / 山火雾霾 / 经济衰退），系统用真实数据给出「扛得住 / 吃紧 / 会崩」的判决，并列出决定因素。评分只作为「对照」出现，用来制造「官方绿灯 vs 压力测试说崩」的冲突。
  - 样板文件：**`resilience_stress_test_concept.html`**（项目根目录，双击即可用浏览器打开；也在 Claude Artifact 里）
  - 关键演示：**Brunei + 食物断供** → 官方 81 分绿灯，判决却是「FRAGILE」，因为本地稻米只有 3,700 吨。「Money on paper isn't food on the table.」
- **优点：** 把「用意」顶到最前；用户不需要懂方法论；能接上项目本来就要做的**公民上报**（红灯 → 「一键上报本地问题」）；反差叙事有记忆点。
- **缺点：** 需要重新设计前台呈现层（不改后端数据/算法）；场景判决逻辑需要定义并保持诚实标注。
- **其他同类范式（也可选）：** ②「区域体检报告」（六角=器官，只显示哪项亮红灯 + 一句话）、③只讲方向的大箭头「在变好/变坏」、④反差揭露式数据新闻。

### 路径 C — 混合（推荐倾向）
- **前台用户：** 路径 B 的场景/故事呈现，**不显示裸分**。
- **后台 / 管理端：** 保留完整评分 + 支柱分解 + 覆盖度 + confidence，供团队内部了解。
- **未来 AI chatbot：** 评分 + 元数据（覆盖度、confidence、data_level）当作 AI 的 **grounding 层**，让 AI 知道「哪些问题我们有数据能答、哪些不能答」——即「客户问的东西到底行不行得通」的判断依据就藏在这套覆盖度里。
- **一句话：** 评分不删，但从「展品」降级为「引擎」——前台展示它算出来的颜色和故事，而不是它本身。

## 4. 顺带说明：Resilience 的「历史趋势」

- 单指标的历史趋势（清洁水、预期寿命、失业、贫困、森林损失…）**已经是真实的**，覆盖 4 地区 2000–2025，在 Regional Details 页可看。
- **Resilience 指数本身的历史趋势目前做不到**：`resilience.json` 只存当前快照；而且指数里的 Education、Energy 两根支柱现在踩在单点数据上，往回推就得编造那两根柱子的历史（违反项目「不造假」原则）。要做真趋势，需要额外 ingest 两条年序列 + 改算法逐年计算。
- 结论：**趋势属于指标层（已做、真），韧性属于快照层——不建议把趋势强行焊到韧性上。** 这一点也可一并问 Supervisor 是否需要。

---

## 5. 建议怎么问 Supervisor

1. **受众定位：** 这个 dashboard 的**主要用户**是大众公民，还是评审/政策/研究者？（这直接决定 A 还是 B/C。）
2. **Resilience 的角色：** 韧性指数是要**直接展示给用户**，还是主要作为**内部信号 / 未来 AI 的判断依据**？
3. **呈现方式：** 是坚持六角图+评分（路径 A），还是转成场景/影响式呈现、评分退后台（路径 B/C）？（可给 supervisor 看 `resilience_stress_test_concept.html` 样板对比。）
4. **历史趋势：** Resilience 是否**必须**展示随时间变化？如果是，才值得投入去补数据、改算法（否则趋势保持在单指标层即可）。
